# 谁上传至OSS

## 前端直传OSS

![image](https://markdown-lai.oss-cn-hangzhou.aliyuncs.com/typora/CAEQYBiBgICKj9ur2hgiIDZhMDRjMzdjN2M0OTRlMzQ5NGVlNjRhZGNmNzg1ODA03963382_20230830144006.372-20260609153237120.svg)

1. **配置OSS**：在控制台创建一个Bucket，用于存储用户上传的文件。同时，为 Bucket 配置跨域资源共享（CORS） 规则，以允许来自 Web 端的跨域请求。
2. **配置服务端：**调用STS服务获取一个临时访问凭证，然后使用临时访问凭证和服务端预设的上传策略（如Bucket名称、目录路径、过期时间等）生成签名授权用户在一定时间内进行文件上传。
3. **配置Web端**：构造HTML表单请求，通过表单提交使用签名将文件上传到OSS。

> 由于使用了临时访问凭证，整个过程中不会泄露业务服务器的长期密钥，保证了文件上传的安全性。

## 后端中转上传OSS

后端中转上传 OSS 时，上传链路实际上是：前端 -> 后端 -> OSS。

所以前端看到的**上传进度**，默认只能看到：前端->后端。而后端 -> OSS，这段进度浏览器天然是看不到的。

因此如果前端想要看到真实的上传进度，推荐使用前端直传 OSS。

你不可能因为一个进度，单独做一个：WebSocket 推送、轮询查询、SSE。



## 两种方案对比

| 维度            | 前端直传（推荐）                   | 后端代传                       |
| --------------- | ---------------------------------- | ------------------------------ |
| **带宽成本**    | 用户→OSS，不走服务器，**省流量费** | 用户→服务器→OSS，**双倍流量**  |
| **服务器压力**  | 几乎无，**可支撑高并发**           | 需要处理大文件流，**易成瓶颈** |
| **上传速度**    | 用户直连OSS，**速度快**            | 多一次转发，**延迟增加**       |
| **安全性**      | 可控（临时签名+权限限制）          | 完全控制，但暴露AK风险在服务端 |
| **实现复杂度**  | 中等（需签名服务+回调）            | 简单（传统文件上传）           |
| **大文件/分片** | 原生支持断点续传                   | 需自行实现，复杂               |
