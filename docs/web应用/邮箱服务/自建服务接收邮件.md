# 自建服务接收邮件

在互联网的规则里，接收邮件的门槛远比发送邮件低。因为接收邮件不需要别人信任你的 IP，你只需要在你的域名解析里告诉全世界：“凡是发往我这个域名的邮件，请送到我的服务器 IP 来”，别人就会送过来。

## MX 记录（Mail Exchanger）

这是自建邮件接收服务的**绝对前提**。你必须拥有一个自己的**域名**（比如 `yourdomain.com`）。

当别人给你发邮件（例如发送到 `admin@yourdomain.com`）时，对方的邮件服务器会去查询你域名的 **MX 记录**。

- **MX 记录的作用**：它就像是域名在互联网上的“收件地址指路牌”。
- **如何配置**：你需要登录你的域名服务商（如阿里云、腾讯云、Cloudflare），添加一条 MX 记录，将它指向你自己的服务器公网 IP。

如果没有配置 MX 记录，全世界的邮件服务器都找不到你的服务器在哪里，你就无法接收到任何邮件。



## 职责分工：A 记录找网站，MX 记录找邮局

在互联网的默认规则里：

- **A 记录（Address）**：是给浏览器（HTTP/HTTPS）看的。当用户在浏览器输入 `yourdomain.com` 时，浏览器默认去找 A 记录指向的 IP 的 **80 或 443 端口**（也就是你的 Vue+Spring Boot 网页）。
- **MX 记录（Mail Exchanger）**：是给**邮件服务器（SMTP）\**看的。当有人给你发邮件到 `admin@yourdomain.com` 时，发件方服务器\**绝对不会**去查 A 记录，而是去查询该域名的 **MX 记录**，寻找专门处理邮件的服务器 IP 和 **25 端口**。

> 即邮箱服务不能默认使用A记录，否则将会服务器造成破坏。就算是同一个IP地址，你依然需要显式指定MX记录，表示我已知道邮件服务指向当前IP。

## 端口

### A. 接收邮件（入站）

当外界的服务器通过 MX 记录把邮件投递给你的服务器时，全球互联网的标准协议雷打不动地使用 **`25` 端口**（SMTP 协议）。

当别人给你发邮件时，流量通过 MX 记录进来。此时，你的服务器上**必须有一个程序在监听 25 端口**（比如你自建的 Postfix、Docker-mailserver，或者用 Java 写的 SMTP 监听程序），否则对方会因为“连接被拒绝（Connection Refused）”而退信。

### B. 发送邮件 / 客户端连接（出站）

当你的 Spring Boot 程序（或 Outlook 客户端）连接邮件服务商去发信时，由于 25 端口不安全且常被云厂商封禁，现在默认改用以下加密端口：

- **`465` 端口**：SMTPS（基于 SSL 加密，主流推荐）。
- **`587` 端口**：SMTP STARTTLS（基于 TLS 加密）。



## 防垃圾邮件

**垃圾邮件（Spam）轰炸**：一旦你的 25 端口暴露在公网上，且 MX 记录公开，全世界的垃圾邮件机器人会在几天内盯上你，疯狂往你的服务器发送诈骗、广告邮件。你必须在代码里或服务器层写极其严格的过滤逻辑（比如检查发件人 SPF、黑名单）。

**协议解析极其痛苦**：邮件的格式（MIME 格式）极其复杂。一封邮件可能包含：纯文本、HTML、内嵌图片、多个不同格式的附件、各种奇怪字符集的编码（UTF-8, GBK, ISO-8859-1）。在 Java 中手动解析这些结构很容易出现乱码或解析失败。

**高可用要求**：对方服务器给你的系统发邮件时，如果你的 Spring Boot 刚好在重启或挂掉了，对方服务器在尝试几次失败后，这封邮件就会被退信（Bounce）并彻底丢失。
