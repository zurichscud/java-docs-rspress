# 场景：代码重构



## 方案一：渐进式重构

如果这个业务未来**还会频繁修改或增加新需求**，最好的办法是趁这次机会将老代码“解耦”。你不需要全盘重构，只需要做**最小化改动**：在老代码的末尾，加上一行事件广播。

1. **定义新事件：** 创建一个 `XxxEvent` 类。
2. **修改老 Service：** 注入 `ApplicationEventPublisher`，在老业务逻辑的最后一行，加上 `publisher.publishEvent(new XxxEvent(...))`。
3. **编写新业务：** 编写全新的 `EventListener` 类来处理新需求。

```java
@Service
public class OldUserService {
    @Autowired private ApplicationEventPublisher eventPublisher; // 1. 新增注入

    @Transactional
    public void oldRegisterMethod(String username) {
        // ... 这里是原本的 500 行老代码（千万别动它们） ...
        System.out.println("原本的老业务执行完毕");

        // 2. 只需要在最后加上这一行
        eventPublisher.publishEvent(new UserRegisteredEvent(username));
    }
}
```

```java
// 3. 新的需求完全写在全新的类里，对老代码零污染
@Component
public class NewFeatureListener {
    @EventListener
    public void handleNewFeature(UserRegisteredEvent event) {
        // 新增的风控、大数据、或者发券逻辑
    }
}
```

## 方案二：编写外层包装 Service

如果老代码是系统核心，逻辑极其复杂，团队**明令禁止修改这个老 Service 类**（哪怕加一行代码都不行），你可以采用“包装设计模式”，在外面套一层。

1. **新建一个门面 Service**（例如 `UserFacadeService` 或 `UserCompositeService`）。
2. 在这个新 Service 中，先调用老的 Service，再调用新需求的方法（或者在这里发布事件）。
3. **把前端（Vue）调用的接口引向这个新的 Service**。

```java
@Service
public class UserFacadeService {
    @Autowired private OldUserService oldUserService; // 老服务
    @Autowired private NewFeatureService newFeatureService; // 新服务

    @Transactional
    public void registerUserComposite(String username) {
        // 1. 调用老代码（完全没动老代码的源码）
        oldUserService.oldRegisterMethod(username); 
        
        // 2. 调用新代码（或者在这里发布事件）
        newFeatureService.executeNewLogic(username); 
    }
}
```
