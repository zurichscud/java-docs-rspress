# 场景：新需求开发

假设公司业务做大了，注册成功后，不仅要发邮件、发优惠券，还要：

1. 接入风控系统，审计用户 IP。
2. 将用户数据同步到大数据的推荐系统。
3. 给邀请人发送积分奖励。
4. 在企业微信/钉钉群里发一个“新用户注册”的运营通知。

## 方案 A：直接写在一个 Service 中

```java
@Service
public class UserService {
    @Autowired private UserRepository userRepository;
    @Autowired private EmailService emailService;
    @Autowired private CouponService couponService;
    @Autowired private RiskService riskService;
    @Autowired private BigDataService bigDataService;
    @Autowired private ScoreService scoreService;
    @Autowired private DingTalkService dingTalkService;

    @Transactional
    public void registerUser(String username, String email) {
        // 1. 核心业务：保存用户
        userRepository.save(new User(username, email));

        // 2. 没完没了的次要业务（副反应）
        emailService.sendWelcomeEmail(email);
        couponService.grantNewUserCoupon(username);
        riskService.auditIP(username);
        bigDataService.syncUser(username);
        scoreService.addInviteScore(username);
        dingTalkService.sendNotification(username);
    }
}
```

缺陷：

- **违背“单一职责原则”（Single Responsibility）：** `UserService` 本职工作是管理用户，现在它却成了全公司的“总管”，要关心邮件怎么发、大数据怎么同步、积分怎么加。

- **代码臃肿，极难维护：** 每次市场部想加一个“注册送XX”的新活动，你都得去修改核心的 `UserService` 类。改动核心类意味着**极高的线上风险**，容易改错。

## 方案 B：使用事件驱动

使用事件驱动后，`UserService` 只做一件事：**宣告“用户注册成功了”这个事实，然后就结束。**

```java
@Service
public class UserService {
    @Autowired private UserRepository userRepository;
    @Autowired private ApplicationEventPublisher eventPublisher;

    @Transactional
    public void registerUser(String username, String email) {
        // 1. 只做核心业务
        userRepository.save(new User(username, email));

        // 2. 广播事件，拍拍屁股走人
        eventPublisher.publishEvent(new UserRegisteredEvent(username, email));
    }
}
```

**极其优雅的解耦（谁需要谁监听）：**

- 负责优惠券的同事，自己写一个 `CouponListener` 监听这个事件。
- 负责大数据的同事，自己写一个 `BigDataListener` 监听这个事件。
- **`UserService` 压根不知道、也不需要知道这世上居然还有“大数据”和“优惠券”服务的存在。**

**极强的扩展性（动态插拔）：**

- 如果下个月活动结束，不需要发优惠券了，你只需要删掉（或注释掉）`CouponListener` 即可，**核心的 `UserService` 代码一个字都不用改**。
- 如果需要接入新的风控系统，直接新建一个 `RiskListener` 类，核心代码同样不需要动。
