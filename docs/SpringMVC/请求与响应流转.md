# 请求与响应流转

以Restful风格的POST请求为例：

```
[前端请求 JSON] 
       │(select HttpMessageConverter)
       ▼
 ┌────────────────────────────────────────────────────────┐
 │ 1. RequestBodyAdvice (请求体拦截)                        │
 └────────────────────────────────────────────────────────┘
       │
       ▼
 [HttpMessageConverter 反序列化为 Java 对象] 
       │
       ▼
 [Controller 业务方法 ]
       │ (select HttpMessageConverter)
       ▼
 ┌────────────────────────────────────────────────────────┐
 │ 2. ResponseBodyAdvice (响应体拦截)                       │
 └────────────────────────────────────────────────────────┘
       │
       ▼
 [HttpMessageConverter 序列化为 JSON] 
       │
       ▼
[Http 响应 JSON]
```

::: tip 总结

1. 选择合适的`HttpMessageConverter`
2. 进入拦截器钩子
3. 执行`HttpMessageConverter`

:::



## ExceptionHandler执行顺序

当你的 `@ExceptionHandler` 方法执行完毕并返回一个对象（例如 `Result.fail("系统异常")`）时，时序如下：

1. **第一步：** `ExceptionHandlerExceptionResolver` 拿到返回值。
2. **第二步：** 路由给对应的处理器（在前后端分离中，通常是 `RequestResponseBodyMethodProcessor`）。
3. **第三步：** 处理器准备把对象转成 JSON。**就在这个关键点，它会先去寻找容器中所有的 `ResponseBodyAdvice`**。
4. **第四步：** 执行 `ResponseBodyAdvice.supports()` 方法，判断当前返回值是否需要拦截。
5. **第五步：** 如果返回 `true`，则执行 `ResponseBodyAdvice.beforeBodyWrite()` 方法。**在这里你可以对 `@ExceptionHandler` 返回的错误对象进行二次修改**（例如统一包装结构、记录日志等）。
6. **第六步：** `ResponseBodyAdvice` 执行完毕，把（修改后的）对象还给处理器。
7. **第七步：** 处理器调用 `HttpMessageConverter`（Jackson）将对象序列化为 JSON 字符串。
8. **第八步：** 写入 `HttpServletResponse` 响应流，返回给 前端。

在 `ResponseBodyAdvice` 的 `supports` 或 `beforeBodyWrite` 中做类型判断，如果发现 `body` 已经是 `Result` 类型（或者是你定义的全局错误对象），就直接返回，不再重复包装：

```java
@Override
public Object beforeBodyWrite(Object body, ...) {
    if (body instanceof Result) {
        return body; // 如果已经是包装好的错误结果，直接放行
    }
    return Result.success(body);
}
```
