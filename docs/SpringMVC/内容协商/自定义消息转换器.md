# 自定义消息转换器

## AbstractHttpMessageConverter

在实际开发中，我们通常不直接去 `implements HttpMessageConverter`，而是选择继承 `AbstractHttpMessageConverter`

原因主要有以下几点：

如果直接实现该接口，你需要自己去写逻辑来判断：

- 当前传入的 `MediaType` 是否匹配？
- 当前支持什么编码？
- 如何处理通配符（如 `application/*`）？

而 Spring 提供的抽象类 `AbstractHttpMessageConverter` 已经帮你把这些通配符匹配、媒体类型检查、字符集（Charset）处理等**底层 HTTP 协议细节全部封装好了**。你只需要专注最核心的两个业务逻辑：

- `readInternal`：怎么把文本反序列化成对象。
- `writeInternal`：怎么把对象序列化成文本。



```java
import com.fasterxml.jackson.dataformat.yaml.YAMLMapper;
import org.springframework.http.MediaType;
import org.springframework.http.converter.json.AbstractJackson2HttpMessageConverter;
import java.nio.charset.StandardCharsets;

public class YamlHttpMessageConverter extends AbstractJackson2HttpMessageConverter{

    public YamlHttpMessageConverter(){
        // 只需要传入 Jackson 的 YAMLMapper，并指定 MediaType 即可
        // 剩下的序列化、反序列化、流读写全部由父类（Jackson 核心槽位）自动搞定
        super(new YAMLMapper(), new MediaType("application", "x-yaml", StandardCharsets.UTF_8));
    }
}
```

## 模板方法（Template Method 模式）

这是它最核心的设计模式。它把复杂的流处理逻辑已经在 `read` 和 `write` 方法中实现了，并对外暴露了两个受保护的抽象方法供子类实现：

- **readInternal**：子类只需要专注从 `inputMessage.getBody()` 的输入流中读取字节，并反序列化为 `clazz` 类型的对象。
- **writeInternal** ：子类只需要专注把 Java 对象 序列化，并写入到 `outputMessage.getBody()` 的输出流中。
