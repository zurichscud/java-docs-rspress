# 鉴权流程

在SpringSecurity中会使用默认的FilterSecurityInterceptor来进行权限校验。在FilterSecurityInterceptor中会从SecurityContextHolder中获取Authentication，然后获取其中的权限信息，校验当前用户是否拥有当前资源所需的权限。

所以我们在项目中只需要把当前用户的权限信息存入Authentication，并设置资源所需要的权限即可

## 核心组件

| **核心组件**                                         | **作用描述**                                                 | **对应类 / 接口**                                        |
| ---------------------------------------------------- | ------------------------------------------------------------ | -------------------------------------------------------- |
| **主体与权限** (Authentication)                      | 当前登录用户是谁，拥有哪些权限（`GrantedAuthority`集合，如 `ROLE_ADMIN`、`sys:user:add`）。 | `SecurityContextHolder.getContext().getAuthentication()` |
| **受保护资源** (FilterInvocation / MethodInvocation) | 用户尝试访问的资源，可以是 **HTTP 接口**（URL）或 **Java 方法**。 | `HttpServletRequest` 或 `MethodInvocation`               |
| **访问决策管理器** (AuthorizationManager)            | 鉴权的真正“执行者”，比对用户权限与资源所需权限，返回同意或拒绝。 | `AuthorizationManager`                                   |



## 鉴权的核心执行流程



**1. 拦截请求 (AuthorizationFilter):** Filter 链最后一环.

请求到达 `AuthorizationFilter`（Spring Security 6+ 推荐，早期版本为 `FilterSecurityInterceptor`）。

**2. 获取用户上下文 (SecurityContext):**提取权限信息.

从 `SecurityContextHolder` 中取出当前的 `Authentication` 对象，拿到用户的权限列表 `getAuthorities()`。

**3. 访问决策 (AuthorizationManager):**核心判定逻辑.

将用户权限与当前接口配置的权限要求（如 `hasRole('ADMIN')` 或 `hasAuthority('sys:user:delete')`）进行比对。

**4. 结果处理 (Decision Handling):**放行或抛出异常.

- **通过：** 触发 `FilterChain.doFilter()`，请求放行到 Controller。
- **拒绝：** 抛出异常并返回 HTTP 403。
