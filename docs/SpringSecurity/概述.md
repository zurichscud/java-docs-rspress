# 概述

## 为什么需要springsecurity

这是一个非常经典且直白的问题。很多开发者在刚接触 Spring Security 时都会有同样的疑惑：“我写一个 `JwtFilter`，解析 Token，校验一下过期时间，然后把用户信息塞进请求上下文，不就大功告成了吗？为什么要引入 Spring Security 这么重的框架？”

你的直觉没错，对于一个**极其简单**的系统，只靠一个 Token Filter 确实能跑通。但随着业务的发展，Spring Security 的真正价值才会逐渐体现。它不仅仅是一个“过滤器”，而是一整套**企业级的安全解决方案**。

## 为什么只用一个 Token Filter 不够？

如果你只用一个 Token Filter，很快就会遇到以下这些不得不自己“重复造轮子”的痛点：

### 1. 复杂的权限控制（鉴权）

Token Filter 通常只负责**认证（Authentication）**——即“你是谁”。但在实际业务中，更难的是**鉴权（Authorization）**——即“你能做什么”。

- **只用 Filter**：你需要在代码里手写很多 `if/else`。例如：`if (!user.getRoles().contains("ADMIN")) { return 403; }`。如果不同接口有不同的权限规则，代码会变得极其臃肿。
- **Spring Security**：提供了优雅的注解（如 `@PreAuthorize("hasRole('ADMIN')")`），可以直接精确到方法级别，甚至支持复杂的表达式（SpEL）。

### 2. 统一的异常处理体系

安全相关的异常处理（如“未登录”和“登录了但没有权限”）需要有统一的国际化规范。

- **只用 Filter**：在 Filter 链中，普通的 Spring MVC `@ControllerAdvice` 往往是**拦截不到** Filter 抛出的异常的。你必须在 Filter 里手动用 `ObjectMapper` 往 `HttpServletResponse` 里写 JSON 字符串。
- **Spring Security**：内置了 异常处理机制，无需手动处理。

### 3. 防范常见的安全漏洞（Web Attack）

一个安全的系统不仅仅只有 Token。

- **只用 Filter**：你很难兼顾到 **CSRF（跨站请求伪造）**、**CORS（跨域资源共享）**、**Session 固定攻击**、**Clickjacking（点击劫持）** 等。
- **Spring Security**：默认开启了多种防护机制，并提供了标准的配置方式（如 `http.cors()` 和 `http.csrf()`）。

### 4. 链路上下文的管理（SecurityContext）

在多线程或异步编程中（比如在 Java 后端使用 `@Async` 异步任务），如何保证当前线程能拿到登录用户的信息？

- **只用 Filter**：如果你自己用 `ThreadLocal` 存用户信息，异步子线程是拿不到的，处理不好还容易导致内存泄漏。
- **Spring Security**：提供了一套成熟的 `SecurityContextHolder` 策略（如 `InheritableThreadLocal`），专门处理多线程环境下的安全上下文传递。



## SpringSecurity兼容性问题

- **如果你在维护老项目（Spring Boot 2.x / Spring Security 5.x）**：过滤器链的最后一环叫 `FilterSecurityInterceptor`，它内部靠 `AccessDecisionManager` 投票来决定权限。
- **如果你在开发新项目（Spring Boot 3.x / Spring Security 6.x）**：最后一环叫 `AuthorizationFilter`，它内部靠 `AuthorizationManager` 直接做权限匹配。

它们在过滤器链中的**生态位和职责是完全一模一样的**，都是负责守住最后一关，做最终的权限校验。
