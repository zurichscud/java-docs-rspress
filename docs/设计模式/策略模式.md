# 策略模式

## 策略模式解决什么问题

假设你在开发一个电商系统的**支付功能**，支持微信支付、支付宝和 PayPal：

不用策略模式的代码（`if-else` 灾难）

```java
public class OrderService {
    public void pay(String payType, double amount) {
        if ("WECHAT".equals(payType)) {
            // 微信支付的调起逻辑、签名认证...
            System.out.println("使用微信支付：" + amount);
        } else if ("ALIPAY".equals(payType)) {
            // 支付宝的 SDK 调用、秘钥解析...
            System.out.println("使用支付宝支付：" + amount);
        } else if ("PAYPAL".equals(payType)) {
            // PayPal 汇率换算、API 呼叫...
            System.out.println("使用 PayPal 支付：" + amount);
        }
    }
}
```

**这种写法的问题在于：**

- **违反开闭原则（OCP）**：如果新增一种“银联支付”，必须修改 `pay` 方法本身，极易引入 bug。
- **臃肿且难维护**：所有支付逻辑堆在一个类里，代码会越来越庞大。

## 策略模式的 3 个核心角色

```java
+-------------------+             +-----------------------+
|  StrategyContext  | ----------->|    PaymentStrategy    |  (策略接口)
| (上下文/Holder)    |             +-----------------------+
+-------------------+                         ^
                                              |
                    +-------------------------+-------------------------+
                    |                                                   |
        +-----------------------+                           +-----------------------+
        | WeChatPaymentStrategy |                           |  AliPayPaymentStrategy| (具体策略)
        +-----------------------+                           +-----------------------+
```

- **抽象策略（Strategy）**：定义所有策略类必须遵循的统一接口。

- **具体策略（Concrete Strategy）**：实现抽象策略接口，编写具体的算法或逻辑。

- **上下文（Context）**：持有策略接口的引用，供客户端调用。

### 第一步：定义抽象策略接口

定义所有策略类必须遵循的统一接口。

```java
public interface PaymentStrategy {
    void pay(double amount);
}
```

### 第二步：实现具体的策略类

实现抽象策略接口，编写具体的算法或逻辑。

```java
// 注册为名为 "wechatPay" 的 Bean
@Component("wechatPay")
public class WeChatPayStrategy implements PayStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("使用微信支付：" + amount + " 元");
    }
}

// 注册为名为 "aliPay" 的 Bean
@Component("aliPay")
public class AliPayStrategy implements PayStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("使用支付宝支付：" + amount + " 元");
    }
}
```

### 第三步：创建上下文（Context）

持有策略接口的引用，供客户端调用。

```java
public class PaymentContext {
    private PaymentStrategy strategy;

    // 允许动态注入/切换策略
    public PaymentContext(PaymentStrategy strategy) {
        this.strategy = strategy;
    }

    public void executePay(double amount) {
        strategy.pay(amount); // 委托给具体策略执行
    }
}
```

### 第四步：客户端使用

```java
@RestController
@RequestMapping("/api/pay")
public class PayController {

    // Spring 会自动将组件映射注入到这个 Map 里：
    // Key: "wechatPay" -> Value: WeChatPayStrategy 实例
    // Key: "aliPay"    -> Value: AliPayStrategy 实例
    private final Map<String, PayStrategy> payStrategyMap;

    // 构造器注入
    public PayController(Map<String, PayStrategy> payStrategyMap) {
        this.payStrategyMap = payStrategyMap;
    }

    @PostMapping
    public String executePay(@RequestParam String payType, @RequestParam double amount) {
        // 1. 根据前端传进来的 payType（如 "wechatPay" 或 "aliPay"）在 Map 中寻找策略
        PayStrategy strategy = payStrategyMap.get(payType);

        // 2. 边界检查：防止传入无效的策略名称
        if (strategy == null) {
            return "支付失败：不支持的支付方式 [" + payType + "]";
        }

        // 3. 执行具体策略（无任何 if-else 分支）
        strategy.pay(amount);

        return "支付指令已发送，方式：" + payType;
    }
}
```

## 对比

**策略模式（Strategy）** 和 **适配器模式（Adapter）** 在结构上确实非常像——它们本质上都是通过**组合（Composition）**，在内部持有另一个对象的引用，并进行委托调用。

但它们设计的出发点（意图）和解决的问题完全不同。用一句话总结它们的根本区别：

> **策略模式是为了“提供多种不同的解法”，而适配器模式是为了“抹平不兼容的接口”。**

### 策略模式：接口是统一设计的

策略模式从一开始就约定好了规范，所有策略实现类都主动去实现这个接口。

```java
// 1. 先定义好统一规范
public interface PayStrategy {
    void pay(double amount);
}

// 2. 大家都按照规范写实现
public class WeChatPay implements PayStrategy { ... }
public class AliPay implements PayStrategy { ... }
```

### 适配器模式：接口是“被迫适配”的

```java
// 这是第三方的类，接口跟你不兼容
public class FastPaySDK {
    public void doDirectPay(BigDecimal decimal) {
        System.out.println("第三方 SDK 支付: " + decimal);
    }
}

// 适配器：强行把它包装成你的 PayStrategy
public class FastPayAdapter implements PayStrategy {
    private FastPaySDK fastPaySDK; // 持有不兼容的对象

    public FastPayAdapter(FastPaySDK fastPaySDK) {
        this.fastPaySDK = fastPaySDK;
    }

    @Override
    public void pay(double amount) {
        // 在内部做参数转换和方法映射
        BigDecimal decimal = BigDecimal.valueOf(amount);
        fastPaySDK.doDirectPay(decimal); 
    }
}
```
